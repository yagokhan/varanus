## Phase 2 — Dual-Engine Architecture

### 2.1 Short Hunter (Frozen)

All short trades (direction == -1) use the frozen Trial #183 parameters:

| Parameter | Value | Note |
|---|---|---|
| `conf_thresh_short` | 0.786 | Entry gate for shorts |

| `leverage_5x_trigger` | 0.968 | Confidence required for 5x lev |

**Implied R:R** = tp / sl = 5.768 / 0.709 = **8.13:1** (high-conviction Hunter setup)

### 2.2 Long Runner (Optimized — v5.2 Deep Search)

All long trades (direction == 1) use a separate, independently-optimized parameter set:

| Parameter | Search Range | Rationale |
|---|---|---|
| `conf_thresh_long` | [0.55, 0.72] | DS: low floor captures moderate-confidence reversals |
| `tp_mult_long` | [2.5, 4.5] | Smaller, "step-like" targets for trending longs |
| `sl_mult_long` | **1.0 (FIXED)** | Hard Stop guardrail — not searched, protects short buffer |

**Training imbalance fix (v5.2.2):** Long arm `scale_pos_weight` = neg/pos per fold.
Without this, XGBoost suppresses p_long scores due to short label dominance (~97% of
training rows are not-long). The per-fold weight correction forces the model to treat
each long candidate as equivalent to `neg/pos` non-long examples, breaking the ~56 signal ceiling.

**Why asymmetric?** Short setups in crypto exploit liquidity sweeps above highs
(sharp, high-conviction breakdowns). Long setups after selloffs are more gradual,
mean-reverting — they need a lower confidence gate, tighter TP target, and a
well-placed stop below the sweep low.

## Phase 3 — HTF Bias Override ("The Oversold Exception") + Volume Injection

### Problem
The 1D MSS (HTF bias) filter blocks long entries when the macro trend is bearish.
In deep selloffs, this creates a **filtering deadlock** — the 4h structure forms
bullish MSS signals but the 1D bias == -1 suppresses them. Previous thresholds
were too restrictive, catching only extreme capitulation events and generating
only ~17 long trades across 5 folds.

### Solution: `bias_bypass_long` Feature (17th) + `ssl_sweep_long` Feature (18th)

**File:** `varanus/pa_features.py`
**SHORT HUNTER LOCK-DOWN:** All changes here are strictly long-side only.

#### `bias_bypass_long` — Aggressive Bias Bypass (v5.2 Volume Injection)

**Config (current — Deep Search):**
```python
BIAS_BYPASS_CONFIG = {
    "rsi_1d_threshold":       50,    # v5.2 Deep Search: 35 → 45 (VI) → 50 (DS)
                                     # Captures mid-range pullbacks and Higher Low
                                     # structures in developing uptrends.
    "price_range_bottom_pct": 0.40,  # v5.2 Deep Search: 10% → 25% (VI) → 40% (DS)
                                     # Enables dip-buying in the lower half of range.
    "range_lookback_4h":      600,   # 100 days × 6 4h bars (unchanged)
    "ssl_sweep_mult_long":    0.80,  # SSL sweep threshold multiplier (unchanged)
    "minor_mss_lookback":     8,     # Deep Search: 8-bar minor bullish MSS detection
}
```

**Bypass Condition:**
```
bypass_active = (RSI_1D < 50) OR (close in bottom 40% of its 100-day H-L range)
```

**What changed (VI → Deep Search):**
- `RSI_1D < 50` (was 45 at VI): captures neutral-regime pullbacks and Higher Low formations
- Bottom 40% of range (was 25% at VI): enables dip-buying in developing uptrend structures
- These thresholds interact with the XGBoost model — it still learns *which* bypass
  bars to trade based on all 19 features combined

#### `ssl_sweep_long` — Sell-Side Liquidity Sweep (18th feature, new)

**What it does:**
Detects downward sweeps of prior swing lows with a **20% relaxed** `min_sweep_pct`
threshold. A bar qualifies if: `low < prior_low × (1 - sweep_min_pct × 0.80)` AND
`close > prior_low` (reversal close).

**Why:**
The standard FVG sweep threshold may miss genuine SSL setups where the wick
is slightly shallower. This feature expands the bypass candidate pool to include
"partial sweep" entries for dip buying, without weakening the full FVG requirement
for the standard (non-bypass) long signal path.

**Effect on `build_dual_labels()`:**
```python
# Before: bypass only if bullish FVG present
bypass_candidates = (bias_bypass_long==1) & (fvg_type==1) & (mss==0)

# After: bypass if bullish FVG OR relaxed SSL sweep
bypass_candidates = (bias_bypass_long==1) & ((fvg_type==1) | (ssl_sweep_long==1)) & (mss==0)
```

**Goal:** Increase long training label count from ~17 → 50–100 across 5 folds,
giving the XGBoost long model sufficient data to learn valid dip-buying patterns.

## Phase 4 — Long Runner Optuna Objective

### Objective Function: Long Runner Score

```
Long Runner Score (per fold) = long_win_rate_pct × max(long_sharpe, 0)

- Maximises both win frequency AND consistency of returns
- Hard gate: long_win_rate < 38% → fold returns -10.0 (penalty, not -999 to preserve pruning)
- DD penalty: if fold MaxDD > 12% → score × 0.40 (same as Hunter)
- Floor: if total long trades < 20 across all folds → return -999.0
```

**Why Win Rate × Sharpe?**
- Win rate alone can be gamed by tiny wins / huge losses
- Sharpe alone can be driven by a few large-pnl outliers
- Their product rewards strategies that win consistently AND generate stable returns

### Search Space (2 Parameters — v5.2 Deep Search)

```python
# SHORT HUNTER LOCK-DOWN: These ranges apply to direction==1 only.
LONG_RUNNER_SEARCH_SPACE = {
    "conf_thresh_long": (0.55, 0.72),   # DS: lowered floor 0.60 → 0.55, ceiling 0.75 → 0.72
                                         # Maximises signal density while maintaining quality gate.
    "tp_mult_long":     (2.5,  4.5),    # Unchanged — step-like targets
    # sl_mult_long: REMOVED from search — FIXED at 1.0x ATR (Hard Stop guardrail)
    # Risk guardrail: 1.0x ATR stop on all long entries protects the $30k+ short buffer.
}
```

**Signal Density Target:** 60–100 long trades across the 5-fold OOS windows.
Optimizer hard-gates trials with < 60 total long trades (`min_total_long_trades=60`).
Baseline (pre-Deep Search): 32 long trades. v5.2.1 WFV result: 49 longs (ceiling hit).
v5.2.2 target: 60+ longs enabled by `scale_pos_weight` per-fold correction.

Short params are injected as frozen constants:

## Phase 5 — Directional Performance Metrics

`compute_metrics()` now returns a full directional breakdown:

```
Portfolio metrics (% returns primary):
  total_return_pct    — Full period return (%)
  cagr_pct            — Annualised return (%)
  max_drawdown_pct    — Peak-to-trough drawdown (%)
  sharpe_ratio        — Annualised Sharpe
  hunter_efficiency   — Net Profit / |MaxDD| (added by walk_forward)

Directional breakdown:
  long_trades         — Count of long trades
  short_trades        — Count of short trades
  long_win_rate_pct   — Win rate for longs (%)
  short_win_rate_pct  — Win rate for shorts (%)
  long_net_pnl_pct    — Long contribution to total return (%)
  short_net_pnl_pct   — Short contribution to total return (%)
  long_sharpe         — Sharpe ratio for long trades only
```
