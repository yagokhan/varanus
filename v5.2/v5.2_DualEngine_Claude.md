### 2.2 Long Runner (Optimized — v5.2)

All long trades (direction == 1) use a separate, independently-optimized parameter set:

| Parameter | Search Range | Rationale |
|---|---|---|
| `conf_thresh_long` | [0.680, 0.850] | Lower gate → higher frequency, catches reversals |
| `tp_mult_long` | [2.5, 4.5] | Smaller, "step-like" targets for trending longs |
| `sl_mult_long` | [0.60, 1.00] | Protective stop tuned for reversal structures |

**Why asymmetric?** Short setups in crypto exploit liquidity sweeps above highs
(sharp, high-conviction breakdowns). Long setups after selloffs are more gradual,
mean-reverting — they need a lower confidence gate, tighter TP target, and a
well-placed stop below the sweep low.

## Phase 3 — HTF Bias Override ("The Oversold Exception")

### Problem
The 1D MSS (HTF bias) filter blocks long entries when the macro trend is bearish.
In deep selloffs, this creates a **filtering deadlock** — the 4h structure forms
bullish MSS signals but the 1D bias == -1 suppresses them.

### Solution: `bias_bypass_long` Feature (17th feature)

**File:** `varanus/pa_features.py`

**Config:**
```python
BIAS_BYPASS_CONFIG = {
    "rsi_1d_threshold":       35,    # Daily RSI below 35 → override
    "price_range_bottom_pct": 0.10,  # Price in bottom 10% of 100-day range → override
    "range_lookback_4h":      600,   # 100 days × 6 4h bars
}
```

**Bypass Condition:**
```
bypass_active = (RSI_1D < 35) OR (close in bottom 10% of its 100-day H-L range)

IF bypass_active AND htf_bias == -1:
    htf_bias_effective = 0   (neutralise bearish filter)
ELSE:
    htf_bias_effective = htf_bias  (unchanged)
```

**What it does:**
- When `RSI_1D < 35`: asset is deeply oversold on the daily → mean-reversion long allowed
- When price in bottom 10% of 100-day range: structurally cheap on a historical basis
- In both cases: the bearish 1D MSS is overridden to **neutral**, not bullish
  (requires 4h MSS to still form — this is not a raw buy signal)
- `bias_bypass_long = 1` is stored as the 17th model feature so XGBoost can
  learn to weight bypass-triggered entries differently

**Goal:** Catch mean-reversion long moves after capitulation events (e.g. AVAX -40% in a month,
RSI_1D = 22) that the original filter would fully block.

## Phase 4 — Long Runner Optuna Objective

### Objective Function: Long Runner Score

```
Long Runner Score (per fold) = long_win_rate_pct × max(long_sharpe, 0)

- Maximises both win frequency AND consistency of returns
- Hard gate: long_win_rate < 38% → fold returns -10.0 (penalty, not -999 to preserve pruning)
- DD penalty: if fold MaxDD > 12% → score × 0.40 (same as Hunter)
- Floor: if total long trades < 20 across all folds → return -999.0
```

**Why Win Rate × Sharpe?**
- Win rate alone can be gamed by tiny wins / huge losses
- Sharpe alone can be driven by a few large-pnl outliers
- Their product rewards strategies that win consistently AND generate stable returns

### Search Space (3 Parameters Only)

```
Portfolio metrics (% returns primary):
  total_return_pct    — Full period return (%)
  cagr_pct            — Annualised return (%)

  long_trades         — Count of long trades
  short_trades        — Count of short trades

  short_win_rate_pct  — Win rate for shorts (%)
  long_net_pnl_pct    — Long contribution to total return (%)
```
